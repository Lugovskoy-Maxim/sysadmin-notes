# Хранение данных и авторизация

## Что реализовано

- Локальные пароли пользователей хешируются `bcrypt` с cost 12.
- Сессионный JWT хранится в `HttpOnly`, `SameSite=Lax`, `Secure` в production cookie.
- Старые Bearer-токены временно поддерживаются для обратной совместимости.
- OAuth использует Authorization Code + PKCE и случайный `state`.
- Поддержаны GitHub, Google и Яндекс ID.
- OAuth access token используется только для получения профиля и не сохраняется.
- В `AuthIdentity` хранится только провайдер, стабильный ID аккаунта, email-снимок, имя и аватар.
- Пользователь видит связанные способы входа в настройках и может отключать их.
- Последний способ входа нельзя удалить, пока не задан пароль или не подключён другой провайдер.
- Привязка нового OAuth-аккаунта подтверждается действующей сессией пользователя.
- Привязка Telegram использует одноразовый случайный deep-link токен со сроком 10 минут.
- В базе хранится только SHA-256 Telegram-токена; повторное использование запрещено.
- Команды бота принимаются только в личном чате от уже привязанного Telegram ID.
- Ссылка `/login` одноразовая, действует 5 минут и после обмена помечается использованной.
- Telegram webhook требует `X-Telegram-Bot-Api-Secret-Token`.
- `password`, `totpSecret` и `sshKey` заметок шифруются AES-256-GCM.
- Ключ шифрования выводится через scrypt из серверного секрета и ID пользователя.

## Модель данных

`User` является внутренним владельцем проектов и заметок. Несколько `AuthIdentity` могут быть
привязаны к одному пользователю. Связь по email создаётся только после получения email от
OAuth-провайдера; для GitHub выбирается primary + verified email.

Токены GitHub, Google и Яндекс не нужны после входа и намеренно не записываются в БД.
Если появятся интеграции, работающие от имени пользователя, их refresh-токены следует хранить
в отдельной таблице, шифровать отдельным KEK и поддерживать отзыв/ротацию.

## Ограничение текущего шифрования

Это server-side encryption, а не zero-knowledge vault: работающий сервер может расшифровать
секреты. Для следующего уровня защиты нужен envelope encryption:

1. Случайный DEK для каждого пользователя.
2. DEK шифруется ключом KMS/Vault и хранится отдельно от данных.
3. Ротация KEK не требует перешифрования всех заметок.
4. Аудит операций расшифрования и аварийный отзыв ключа пользователя.

## Production checklist

- Задать независимые `JWT_SECRET` и `VAULT_ENCRYPTION_SECRET` длиной не менее 32 случайных байт.
- Использовать PostgreSQL с шифрованными дисками и зашифрованными резервными копиями.
- Не логировать тела запросов, cookies, OAuth codes и поля заметок.
- Ограничить CORS одним `FRONTEND_URL`; использовать только HTTPS.
- Зарегистрировать точные callback URL из `backend/.env.example`.
- Добавить CSP, лимит активных сессий, отзыв всех сессий и журнал входов.
- Не создавать OAuth-связь по неподтверждённому email.
- Периодически проверять восстановление резервной копии и ротацию ключей.

## Импорт менеджеров паролей

Поддерживаются Bitwarden JSON, KeePass XML/CSV, 1Password CSV, LastPass CSV,
Chrome CSV и Firefox CSV. Разбор выполняется в браузере. На сервер отправляются только
подтверждённые пользователем записи, после чего секретные поля шифруются.

Экспорт менеджера паролей обычно является открытым текстом. Его следует удалить сразу после
импорта и не загружать в облачные диски, почту или систему контроля версий.
